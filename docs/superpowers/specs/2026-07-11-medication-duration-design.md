# 복약 기간(duration) 설계 — 슬롯별 발생일 롤링 모델

- **작성일**: 2026-07-11
- **대상 버전**: 1.1.0 (기능 릴리즈)
- **브랜치**: `feature/medication-duration`

## 1. 배경 / 문제

현재 약은 `endDate`(달력 날짜)로 복용 종료를 관리하는데, **알람 스케줄이 종료일을 전혀 안 지킴**:

- `notification_service._scheduleSlotAlarms`가 `matchDateTimeComponents: DateTimeComponents.time`로 **매일 무한 반복** 등록 (endDate 체크 없음)
- endDate는 앱을 열어 `rescheduleAllAlarms` 할 때만 걸러짐 → **종료일 지난 뒤 앱을 안 열면 알람이 계속 울림**
- 실기기 확인(2026-07-11): "1일치" 약이 종료일 다음날 08:00에도 발동. dumpsys에 다음날 알람이 예약돼 있음.

복약앱에서 **끝난 약을 계속 먹으라고 알리는 건 위험**(안 먹어도 될 약 재복용).

## 2. 결정된 동작 (사용자 확정)

"복용 종료일(날짜)" 개념을 버리고 **"복용 기간(일수)"** 으로 전환. 핵심 규칙:

> **각 슬롯(오전/오후/저녁)이 정확히 N번 발동한다. 등록 시점에 이미 지난 슬롯은 다음날로 롤오버된다.**

예) 오전 08:00 / 오후 13:00 / 저녁 19:00, **1일치**를 오늘 12:30에 등록:

```
오늘 13:00  오후 ✓
오늘 19:00  저녁 ✓
내일 08:00  오전 ✓   (오전은 이미 지나서 내일로)
→ 총 3회 후 종료
```

- `durationDays == null` → **계속(무한)**, 매일 반복 (만성 복용약)
- 병원에서 낮에 약 받아 등록해도 하루치를 온전히 다 챙긴다.

## 3. 데이터 모델

`lib/models/medicine.dart`

- **제거**: `endDate` (DateTime?)
- **추가**: `durationDays` (int?, `null` = 계속)
- **유지**: `startDate` (등록 시각 = 발생일 계산 기준점)
- Firestore 필드: `durationDays` (int?), `endDate` 필드 미사용

## 4. 핵심 규칙 — 슬롯 발생일 계산 (단일 소스)

한 곳에 순수 함수로 (테스트 가능하게). 대략:

```
slotOccurrences(startDate, "HH:MM", durationDays) -> List<DateTime>
  first = 그 시각의 첫 발생 datetime that >= startDate
          (오늘 시각이 지났으면 내일 그 시각)
  durationDays == null -> 무한 (윈도우 범위까지만 실제 생성)
  else -> [first, first+1d, ..., first+(N-1)d]
```

- 모든 스케줄·표시가 이 함수 하나를 공유 → 알람/홈/로그 일관.
- 기기 시각(`kstNow`) 사용 → 오프라인에서도 정확.

## 5. 알람 스케줄 — 7일 롤링 윈도우

`notification_service`

- **`matchDateTimeComponents.time`(무한 반복) 폐기.**
- 대신 **앞으로 7일치**를 **(날짜, 슬롯시각) 단위로 개별 스케줄**:
  - 각 날짜/슬롯에 대해 그날 활성인 약들을 모아 **1개 알람**(기존처럼 슬롯 묶음 유지: "약A, 약B")
  - 활성 판정 = 규칙4 (무한약=매일, 기간약=발생일에만)
- **윈도우 갱신 시점**: 앱 열 때 + 부팅 시(기존) + **알람 "복용완료" 처리 시**(윈도우 앞으로 재연장)
  - → 무한약도 7일 윈도우가 계속 앞으로 밀려 끊기지 않음. 매일 쓰는 앱이라 자연스럽게 유지.
  - 알려진 한계: 앱을 7일 넘게 한 번도 안 열고 알람 처리도 안 하면 그 이후 알람 없음(극단적, 실사용에선 거의 없음).
- **+10분 리마인더**도 동일하게 (날짜, 슬롯) 단위로.

## 6. 알람 ID / 취소 (과거 중복-고아 버그 방지)

- **ID**: (날짜오프셋, 슬롯시각) 인코딩. 리마인더는 별도 오프셋 범위. (정확한 인코딩은 구현 시 확정, int32 안전 범위)
- **취소**: 스케줄한 알람 ID 집합을 `shared_preferences`에 저장 → 재등록 시 **저장된 ID 전부 취소 후 재생성**.
  - 기존 `cancelAllSlotAlarms`가 캐시된 슬롯시각만 취소해 고아가 남던 문제(2026-07-04 진단)를 근본 차단.
  - 병원 알람·사용자 스누즈는 기존대로 보존.

## 7. 로그 / 홈 표시

`firestore_service` 로그 생성부

- 현재 `[startDay ~ endDay]` 달력 체크 → **규칙4(그 슬롯이 그 날 활성인가)** 로 교체.
- 홈 주간 달력은 **로그 기반**이라 자동 반영 (홈 화면 코드 거의 불변).
- 홈에서 다음 주로 넘겨봐도 로그는 날짜별 on-demand 생성 → 7일 윈도우와 무관하게 정상 표시.
- **7일 윈도우는 "알람 등록 범위"일 뿐 "화면 표시 범위"가 아님.** 홈에서 몇 주·몇 달 뒤로 넘겨도 그 날짜 접근 시 로그 생성 → **무한약(durationDays=null)은 미래 어느 날짜든 계속 표시**. (firestore_service line 240 주석: "미래 날짜 로그는 해당 날짜 접근 시 자동 재생성됨" — 기존 동작 유지)

## 8. UI — 복용 기간 입력 (프리셋 버튼)

`senior_medicine_add_screen`, `senior_medicine_edit_screen`

- **날짜 피커 제거** → 프리셋 버튼:
  - `[1일] [3일] [1주일] [2주일] [계속]`
  - `[직접입력]` → 누르면 인라인 **± 스테퍼** 펼침: `[－]  N 일  [＋]` (큰 버튼, 앱 기존 시간조절 ± 패턴과 통일, 키패드 없음)
- 상세화면(`senior_medicine_detail_screen`): "복용 3일 (~7/14까지)" 또는 "계속"

## 9. 오프라인

- 새 로직 전부 **로컬**(발생일 계산 + `zonedSchedule` + 윈도우 갱신) → 네트워크 불필요.
- 오프라인 등록 → 캐시 저장(기존 offline add) → 즉시 로컬 스케줄. 온라인 복귀 시 Firestore sync(보호자 반영)는 기존과 동일.
- 시니어 본인 알람은 네트워크 무관하게 발동.

## 10. 마이그레이션 / 버전

- 출시 전(프로덕션 미게시)이라 **깨끗한 컷오버**: `durationDays` 없는 기존 doc = `null`(계속)로 취급.
- 버전 **1.0.1 → 1.1.0** (기능 추가).

## 11. 테스트 계획

- **단위 테스트** (슬롯 발생일 계산):
  - 등록 시각이 슬롯 이전/이후 경계 (롤오버 유무)
  - N=1 / N=3 / null(무한→윈도우 경계)
  - 자정 근처 경계
- **실기기**:
  - 기간약: N회 후 정확히 멈춤 (다음날 알람 없음)
  - 무한약: 지속 (윈도우 갱신으로 이어짐)
  - 롤오버: 낮에 1일치 등록 → 오후·저녁 오늘 + 오전 내일
  - 중복 없음 (dumpsys alarm 카운트 확인)
  - **오프라인 등록 → 알람 정상 발동**

## 13. 범위 밖 (비목표)

- 안드14 풀스크린 권한 온보딩 유도 (별개 작업, 사용자가 "직접 켜면 됨"으로 당장 보류)
- 복용 시작일을 미래로 지정하는 기능 (현재 startDate = 등록 시각 고정)
- 요일별/격일 복용 등 복잡한 스케줄 (이번 범위 아님)
- **보호자-오프라인 확인 문제**(2026-07-11 사용자 제기): 동거 가족이 있고 시니어가 데이터/와이파이를 안 켜면, 복용 로그가 로컬 캐시에만 있어 **보호자 화면에 동기화 안 됨** → 보호자가 복용 여부 확인 불가. 이건 기존 **B6**(오프라인 복용 시 보호자 거짓 미복용 알림)와 같은 뿌리. **이번 duration 작업과 별개, 추후 presence/동기화 설계로 다룸.**

## 14. 별건 추후 튜닝 (이번 작업 중 발견, duration과 무관)
